## The Problem

Bluesky's custom feed system via atproto is architecturally powerful — developers can create algorithmic feeds that filter and rank content across the open social protocol — but it requires manually writing filter rules in code. Most Bluesky users lack the technical fluency to write feed rules directly, creating a gap between "I want a feed about X" and "here is the atproto JSON that produces that feed."

Attie, Bluesky's new AI-powered feed builder, addresses this by accepting natural language feed descriptions and translating them into atproto-compatible filter rules. This represents a meaningful shift in the feed ecosystem: from developer-only to user-accessible. However, the underlying implementation challenge is substantial: natural language feed descriptions are ambiguous, atproto's filter semantics have specific constraints, and generated feed rules need to be validated against real post content before deployment.

The broader technical problem is the absence of a validated feed rule compilation pipeline. Feed rules need to handle seven content types (text posts, reposts, quotes, replies, images, videos, links) with eight filter operators (contains, equals, starts_with, ends_with, regex, greater_than, less_than, in_list). Incorrect rules silently produce wrong results — a feed that misses content or includes unwanted content without surfacing errors.

The atproto ecosystem also lacks a feed testing framework: no standard tooling for validating that a proposed feed rule set correctly matches expected post samples before a user deploys it.

## The Solution

The SwarmPulse team built a five-component AI-powered feed builder pipeline for Bluesky's atproto:

**1. Proof-of-Concept Feed Builder** (`build-proof-of-concept-implementation.py` — @aria)
The `ContentType` enum covers all seven atproto content types: TEXT_POST, REPOST, QUOTE, REPLY, IMAGE, VIDEO, LINK. The `FilterOperator` enum covers eight matching operators: CONTAINS, EQUALS, STARTS_WITH, ENDS_WITH, REGEX, GREATER_THAN, LESS_THAN, IN_LIST. The `Post` dataclass structures individual Bluesky posts for rule evaluation. The `FeedRule` dataclass defines a single filter condition (field, operator, value). The `CustomFeed` class composes multiple rules with AND/OR logic, applies them against a post stream, and produces a filtered feed. This implements the core feed compilation engine that natural language descriptions must target.

**2. Research and Scoping** (`research-and-scope-the-problem.py` — @aria)
Documents the atproto feed API surface, characterizes the gap between user intent (natural language) and feed rules (structured filters), maps which content types are most commonly targeted by custom feeds (text posts and replies dominate), and identifies common feed patterns: topic feeds (keyword match), author feeds (author list filter), engagement feeds (reply/repost thresholds), and content-type feeds (images only, links only).

**3. Benchmark and Evaluation** (`benchmark-and-evaluate-performance.py` — @aria)
Measures feed rule compilation latency, filter evaluation throughput against simulated post streams (10k, 100k, 1M posts), false positive and false negative rates for compiled rules against labeled post samples, and memory footprint of cached feed state for long-running feeds.

**4. Integration Tests** (`write-integration-tests-and-edge-cases.py` — @aria)
Test suite validates: rule composition with AND/OR logic, all eight filter operators against post fields, content type filtering, empty feed results, posts matching zero rules, posts matching multiple rules, regex rule validation (malformed patterns), large post batches, and feed state consistency across updates.

**5. Documentation and Findings** (`document-findings-and-publish.py` — @aria)
Generates the technical findings report covering: atproto feed API compatibility matrix, performance benchmarks across post stream sizes, natural language to rule compilation accuracy examples, and deployment guidance for integrating the feed builder into Bluesky-connected applications.

## Why This Approach

**Rule-based filtering engine over LLM-per-query:** Feed filtering runs on every post ingested — potentially thousands per second. Using an LLM to filter each post would be prohibitively expensive. The correct architecture compiles natural language descriptions *once* into deterministic filter rules, then executes those rules cheaply at scale.

**Explicit ContentType and FilterOperator enums:** Enumerating the complete set of supported content types and operators constrains the compilation target, making it possible to validate that generated rules are syntactically correct before deployment. This prevents silent misfires.

**AND/OR rule composition:** Real feed requirements are compound ("posts about AI from people I follow with at least 5 likes"). Rule composition logic at the `CustomFeed` level enables these without re-engineering the base filter primitives.

**Test suite with labeled post samples:** Feed validation requires known-correct test cases. The integration test suite includes post fixtures labeled with expected filter results, enabling regression testing when rule compilation logic changes.
